PROJECT: BizConnect Journal

Overview

BizConnect Journal is a dedicated application catered to busy businessmen, especially those who travel a lot. This app helps to manage their wide range of contacts, events and tasks. It also uses Command Line Interface (CLI) as the primary source of input, with visual graphics as additional supported functionality for users.

The everyday life of a businessman can be packed with many events, tasks and network contacts, thus having a single tool to manage all their necessary work information makes things easier, as well as save them valuable time and effort. With these goals in mind, BizConnect Journal is designed to provide one central application without the trouble of having different applications for each individual purpose.

Contributions to the User Guide

Given below are sections I contributed to the User Guide. They showcase my ability to write documentation targeting end-users.

Exporting addressbook data: export [since v1.1]

Exports addressbook data into an XML or CSV file.
Format: export FILEPATH [MORE_KEYWORDS]

  • Addressbook creates XML or CSV file if valid filepath and filename is given.

  • If filename already exists, a warning message is displayed.

  • If file extension is not .xml nor .csv, a warning message is displayed.

  • If filepath not given, the file is auto-generated in the same filepath as the JAR file.

  • If filepath is not found, Addressbook creates folders to support the filepath defined by the user.

Examples:

  • export C:\Users\John Doe\Documents\addressbook.xml
    Creates XML file in the user-defined filepath

  • export C:\Users\John Doe\Documents\addressbook.csv
    Creates CSV file in the user-defined filepath

  • export addressbook.xml
    Creates XML file in the same folder as the JAR file

Adding an event: addevent [since v1.3]

Adds an event into event book.
Format: addevent et/TITLE ed/DESCRIPTION el/LOCATION edt/DATETIME If arguments like TITLE or DESCRIPTION are too long, it may be cut off in the event list. Event details should not be too wordy!

  • Adds an event if all fields have correct inputs.

  • If there is an empty field, event is not added.

  • If datetime field is detected to be incorrect, no event is added.

Examples:

  • addevent et/Christmas Party ed/Celebrating Christmas 2018 el/Samuel’s House edt/2018-12-25 1830
    Creates an event with the corresponding details and adds it into the event book; displays it on event list.

Deleting an event: deleteevent [since v1.3]

Deletes an event in the event book.
Format: deleteevent INDEX

  • Deletes an event if given index is valid.

  • Does not delete event if given index is invalid.

Examples:

  • deleteevent 2
    Deletes the second event on the displayed event list.

Finding an event: findevent [since v1.4]

Shows a list of all events matching the given keywords.
Format: findevent et/KEYWORD [MORE_KEYWORDS]…​ for title OR ed/KEYWORD [MORE_KEYWORDS]…​ for description For DATETIME, it will try to match datetimes containing the input argument, so try to be as accurate as possible!

Example: * findevent edt/01
Returns all events with the argument '01' inside the datetime field. This includes dates like 2018-XX-XX, 2017-XX-XX since '01' is inside '2018' and '2017'.

Listing all events: listallevents [since v1.4]

Shows a list of all events in the address book.
Format: listallevents

Switches between list tabs: switchtab [since v1.4]

Switches between the Events and Tasks list tabs.
Format: switchtab

Jumps to specified year and month: jumpto [since v1.5rc]

Jumps to specified year and month on the calendar.
Format: jumpto yyyy-mm

  • Calendar shows year and month input if valid.

  • Does not change calendar view if given input is invalid.

Examples:

  • jumpto 2018-05
    Jumps to 2018 May on the calendar view.

Editing an event [coming in v2.0]

Edits an existing event in the event list.
Format: editevent 3 et/NEW_TITLE edt/NEW_DATETIME

  • Edits event is given index and prefix is valid

  • Error message if given index or prefix is invalid

Export feature

Current Implementation

It makes use of ExportCommand which extends UndoableCommand. When the Export functionality is called on the AddressBookParser, it parses the given input as a FILEPATH using ExportCommandParser which eventually creates the ExportCommand instance. The ModelManager then invokes methods involving Storage under the Model instance in order to store the data into a user-defined location. If such a location was not well-defined, then the XML file exported will be directly saved in your local repository’s main folder. The 'StorageManager' will then call the exportAddressBook method under XmlAddressBookStorage to store the data as an XML file into a user-defined location.

Design Considerations

Aspect: Usages of Exported Data

  • Alternative 1(current choice): Allows option of exporting data into either CSV or XML files.

    • Pros: Allows greater flexibility as CSV files can be catered to importing of contacts into Google Contacts.

    • Cons: Need to manually cater to popular file formats and this may require constant maintenance on the developers' end.

  • Alternative 2: Only allows exporting of data as XML files.

    • Pros: Simplified export into a file that is readable and understandable by users, is completely compatible with Java and 100% portable.

    • Cons: Users may not know the benefits or uses of an XML file.

Add Event feature

Current Implementation

The add event feature allows the user to add a new event into the Event Book. The event details required: Title, Description, Location, Datetime. All of the fields are compulsory, and the user can simply fill in a dummy value for irrelevant fields.

The following sequence diagram shows how the add event command works under LogicManager.

  1. User executes addevent command with the required arguments.

  2. AddressBookParser class creates a new AddEventCommandParser object to parse the command arguments.

  3. This AddEventCommandParser object parses the arguments and create a new AddEventCommand object.

  4. The addevent command is then executed and handled by the ModelManager class.

  5. The event is finally added to the Event list in Address Book and the updated list is then displayed to the user.

Find Event feature

Current Implementation

The find event feature allows the user to search all events in the Event Book for a particular keyword. The user specifies the field to search under (Title, Location, Datetime) and the keyword argument(s).

  1. User executes findevent command with the required arguments.

  2. AddressBookParser class creates a new FindEventCommandParser object to parse the command arguments.

  3. This FindEventCommandParser object parses the arguments and create a new FindEventCommand object.

  4. The findevent command is then executed and handled by the ModelManager class.

  5. Events with matching keywords under the specified field will be updated in the displayed event list and shown to the user.

Jump To Year-Month feature

Current Implementation

Aside from using the mouse cursor, users can jump to a particular year-month using the CLI. This saves a lot of time, especially when the user wishes to jump a few months back or forth.

Format: jumpto yyyy-mm

  1. User executes jumpto command with the required arguments.

  2. AddressBookParser class creates a new JumpToCommandParser object to parse the command arguments.

  3. This JumpToCommandParser object parses the arguments and create a new JumpToCommand object.

  4. The jumpto command is then executed by creating a new JumpToCalendarRequestEvent that triggers the EventsCenter.

  5. EventsCenter dispatches the event to be handled by CalendarView because it has to populateCalendar method that displays the calendar by taking YearMonth as an input.

  6. popularCalendar finally displays the new CalendarView state to the user.

Switch Tab feature

Current Implementation

The switch tab functionality allows the user to use CLI to switch between the Event and Task tabs as shown below. This provides additional support for users who prefer CLI to mouse controls.

Format: switchtab

And this is how SwitchTab works!

  1. The user executes the switchtab command.

  2. The SwitchTabCommand class creates a new SingleSelectionModel<Tab> object using the TabPane displaying Events and Tasks list.

  3. The selected tab-index for the SingleSelectionModel<Tab> object is updated and displayed to the user.
